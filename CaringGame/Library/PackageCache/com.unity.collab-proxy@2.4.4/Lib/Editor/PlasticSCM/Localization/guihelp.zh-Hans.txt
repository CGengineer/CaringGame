== PlasticPendingChangesMovedOnlyFiles ==

找更改用了太：{totaltime}。

操作超快，因此在速度很慢，我会担心有。

造成情况的一些可能原因如下：

== PlasticPendingChangesMerge ==
[title]有一个合并已准好入[/title]

知道合并程中状的含？

会注意到，合并的文件被“状/从更集号合并”，可区于可能具有的其他本地更改。

* [bold]已出[/bold]：文件已合并，因已同文件行了修改。合并可能是自的，但仍然行了合并。

* [bold]已替[/bold]：文件已存在于的分支中，但是已被新的版本替。未生冲突。

* [bold]已制（新）[/bold]：文件先前在的分支中不存在，是在合并程中建的（从合并源生）。

* [bold]已除[/bold]：合并期除了文件。

有于将在入建的合并接的信息。合并接可以跟踪生的合并操作。

== PlasticPendingChangesMergeEmpty ==
[title]此合并中没有更改，但是...[/title]

遇到了特殊情况。

正在合并，但是已有的文件与合并的文件匹配。

因此，除了合并接之外，没有任何其他要入的内容。

合并接非常重要，因即使未修改文件，也可以通方式将此合并完成。跟踪合并需要使用此接。

== PlasticPendingChangesAllMergeChangesRequired ==
[title]合并在一起的文件，必一起入[/title]

Unity VCS 和所有代分布式版本控制系都以方式工作：合并分支会完全合并分支。

是因 Unity VCS 需要将分支完全合并。不能只合并几个文件，然后在以后重循来合并漏的文件。

原因是合并接是在更集之置的，而不是在个文件之置的。
大幅提高了合并性能，但代价是牲了个文件的活性。

== PlasticPendingChangesMergeSingleBranch ==
[title]，是什？[/title]

不要慌！在分支上工作完成了合并。我解一下生了什。

如果只是合并了几个文件，什所有其他些文件都示了出来？

很。当合并到分支的部，会引入其他人所做的更改。

必合并的文件在“状”列中[bold]已出[/bold]。

所有其他文件：

* [bold]已替[/bold]：已将一个文件从部制到的工作区。

* [bold]已制（新）[/bold]：已将一个没有的新文件制到的工作区。

* [bold]已除[/bold]：文件不再存在于部，因此也已从的工作区中除。

在感好些了？:-)

== CodeReviewWindowGeneralHelp ==
[title]像人士一行！[/title]

我必[link:https://www.plasticscm.com/book/#_review_each_task][/link]任。我一概念了。

[bold]如何行出色的[/bold]
1.模式：“逐个更集”或“完整分支”。下面的信息。
2.可以求更改和/或向作者提。
3.最后，置状：“需要返工”或“已”。

[bold]“逐个更集”和“完整分支”[/bold]
-“完整分支”完整分支行差比。是大家都知道的典型。
-“逐个更集”将遍个更改。在遵循当准的情况下，的效果好。作者将解个更改，明背景信息，助跟随他的思路。好于令人崩的 200 个更改。

在[link:https://www.plasticscm.com/book/#_checkin_often_and_keep_reviewers_in_mind]此[/link]全面了解如何在入更改将者牢于心。

[bold]如何添加注//求更改[/bold]
在差比面板的右，可找到有“+”并可点的垂直条。即可。

== CodeReviewWindowNewComment ==
[title]求更改、提[/title]

可以：

1.[bold]求更改[/bold]：当在批准前需要作者行某更改。Unity VCS 会跟踪是否用了更改，有助于化下一个周期。

2.[bold]提[/bold]：当在批准前需要答案。Unity VCS 会跟踪是否回答了。

可以使用常的旧有注，但更改和聚焦于最重要的方面：化流程，因此了解回答了什或行了什更改将不再重要。

== CodeReviewWindowWithPendingComments ==
[title]回答并解决更改求[/title]

1.看更改求和列表。双可看信息。
2.要解决更改求：在入注中引用更改求的 GUID。看下面的示例。可以利用“待定更改”中的合列表来填写信息。
3.通直接回来回答所有待回的。
4.最后，完成后，将代状更改“已”。

[bold]如何用/更改求[/bold]
要更改求，只需在入注中使用更改求 GUID。
有模式：
* ：[bold][apply-change:516a2741-9069-4c08-983f-71149f0a93c8][/bold]
* 短：[bold][apply-change:516a2741][/bold]（只需指定 GUID 的第一部分）

== GluonWkExplExplainUpdate ==
[title]有新版本可供下[/title]

使用[action:GluonUpdateWorkspace]更新[/action]可下工作区中文件的所有新版本。

当文件有新版本，相的状列将示“期”。可以右文件来文件行独更新，而不是像使用“更新”按那整个工作区行更新。

[bold]提示[/bold]：什候下新文件？假有人向存添加了新文件。“更新”会下文件？如果文件的目配置“完全”，会下文件。但是，如果配置部分，不会下。什？因“部分”意味着只想下某些文件。因此，情况下不会自下所有新的文件。

== GluonConfigurePreviousConfig ==
[title]微的工作区[/title]

* 色勾号：配置要下的新文件或目。
* 色勾号：取消配置某个元素。
* 黑色勾号：已配置待下的内容。
* 无勾号：未配置下到工作区。
* 目上的部分勾号：意味着并非目中的所有内容都会下。如果只了几个文件，就会生情况。


于大的存，下需要理的内容将非常有助，因可以省大量空和下。

[bold]提示[/bold]：[action:GluonConfigSearch]搜索[/action]可找文件。
